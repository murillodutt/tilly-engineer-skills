# Flash-Fry Skill Gap Spec

This spec defines the proposed repair for the Flash-Fry invocation gap in TES. It is a roadmap-level proposal, not yet the active package contract.

## Goal

Make Flash-Fry a project-local installed TES skill surface without changing the meaning of the Mantra Gate marker.

The desired result is:

```text
/flash-fry -> project-local TES skill -> Mantra Gate compact pre-action check
```

The repair must work from installed project runtime surfaces such as `.claude/skills/**` and `.agents/skills/**`. It must not depend on a global skill repository or a Claude/Codex plugin install.

## Problem

TES already has a working Mantra Gate marker:

```text
[🍳 Flash-Fry]
```

The marker is generated by the Mantra Gate helper and described by the active discipline skills. However, the installed project runtime does not define a dedicated skill named `flash-fry`.

This creates a user-facing gap:

1. The operator treats Flash-Fry as an available operational capability.
2. A slash command or skill invocation attempts to call `tilly:flash-fry`.
3. The runtime cannot find that skill.
4. The agent falls back to explanation instead of executing the intended gate.

The failure is not that the marker is unavailable. The failure is that the invocation surface is not materialized as an installed project skill.

## Current Documentation Alignment

This spec aligns Flash-Fry with the current skill models documented by Claude Code, the open Agent Skills specification, and Codex skill guidance:

- Claude Code project skills live at `.claude/skills/<skill-name>/SKILL.md`, and the directory name becomes the direct slash invocation `/skill-name`.
- Agent Skills require a directory containing `SKILL.md`; the `name` field must match the parent directory.
- Codex guidance distinguishes plugins from skills: plugins connect external tools or sources, while skills encode a process or playbook.

Therefore the primary portable contract is a project-local skill named `flash-fry`, not a plugin namespace and not a `tes-flash-fry` primary folder. The `tes-flash-fry` spelling may remain only as an optional compatibility alias if TES chooses to materialize aliases for older operator muscle memory.

Reference docs:

- Claude Code skills: <https://code.claude.com/docs/en/skills>
- Agent Skills specification: <https://agentskills.io/specification>
- Codex plugins and skills: <https://openai.com/academy/codex-plugins-and-skills/>

## Source Boundary

Reference projects may expose the symptom, but they do not define TES behavior. Portable behavior must be defined and implemented only in the TES package source:

- `src/adapters/codex/**`
- `src/adapters/claude/**`
- `src/adapters/cursor/**` when Cursor routing text changes
- `scripts/**` for materialization and oracles
- `docs/**` for governed explanation

Installed target mirrors such as `.claude/skills/**`, `.agents/skills/**`, `.tes/bin/**`, and target bootloaders are certification targets only.

## Terms

| Term | Meaning |
|------|---------|
| Flash-Fry marker | The literal visible marker `[🍳 Flash-Fry]`. |
| Mantra Gate | The governing pre-action contract and helper scripts. |
| `flash-fry` | Proposed installed skill that routes invocation to Mantra Gate. |
| `/flash-fry` | Native slash trigger for the installed `flash-fry` skill. |
| `tes-flash-fry` | Optional compatibility alias, not the primary skill identity. |
| `tilly:flash-fry` | Plugin namespace form; invalid for project-local skills unless a host explicitly supports such routing. |

Flash-Fry must remain a compact Mantra Gate path. It must not become a second governance system.

## Non-Goals

- Do not create or require a global TES plugin.
- Do not depend on Claude global skill installation.
- Do not define behavior from a target project.
- Do not duplicate the full Mantra Gate schema inside a new skill.
- Do not bypass `mantra_gate.py` or `mantra_gate_adoption_oracle.py`.
- Do not make `/flash-fry` perform arbitrary writes.
- Do not change the marker text.

## Proposed Contract

Create a new installed skill named `flash-fry` for Codex and Claude adapters.

The skill is a small router:

1. Recognize explicit invocations:
   - `/flash-fry`
   - `flash-fry`
   - `Flash-Fry`
   - `tes flash-fry`
   - `/tes-flash-fry` when an alias is intentionally materialized
   - `/tes:flash-fry` only as normalized operator intent, not as a native skill namespace
2. Classify the requested mode:
   - `marker`: print the compact marker only.
   - `check`: run the read-only Mantra Gate health/adoption check.
   - `pre-action`: print the marker and name the intended next state-changing action.
   - `audit`: run or instruct the explicit audit-history path.
3. Delegate to existing helpers:
   - `.tes/bin/mantra_gate.py emit-marker`
   - `.tes/bin/mantra_gate.py --self-test`
   - `.tes/bin/mantra_gate_adoption_oracle.py --target .`
   - `.tes/bin/mantra_gate_adoption_oracle.py --target . --audit-history`
4. Stop on `BLOCKED`, `NEEDS_REVIEW`, or `BYPASS_SUSPECTED` when the invocation is tied to closure, commit, push, release, destructive work, public surface, generated runtime packaging, production, secrets, auth, compliance, or data.

The skill may explain what it did, but its primary output is operational:

```text
[🍳 Flash-Fry]
status: PASS|OK|NEEDS_REVIEW|BLOCKED|BYPASS_SUSPECTED
next: <short action or blocker>
```

## Platform Behavior

### Claude Code

Claude must consume the skill from the project-local install:

```text
.claude/skills/flash-fry/SKILL.md
```

Because Claude Code maps project skill directory names to slash invocations, `/flash-fry` should be native when the project-local skill exists. If Claude reports `/flash-fry` as invalid, TES should classify that as an installation, watcher, or routing failure rather than requiring a plugin.

The contract must not require:

- `~/.claude/skills/**`
- `~/.claude/plugins/**`
- `installed_plugins.json`
- a `tilly:*` plugin namespace

If a Claude transcript shows `Skill {"skill":"tilly:flash-fry"}`, TES should classify that as a plugin-namespace routing mismatch. The repair is to route through `/flash-fry` and `.claude/skills/flash-fry/SKILL.md`, not to require a global namespace.

### Codex

Codex must consume the skill from the project-local adapter install:

```text
.agents/skills/flash-fry/SKILL.md
```

The source counterpart lives in:

```text
src/adapters/codex/skills/flash-fry/SKILL.md
```

Codex may already show `[🍳 Flash-Fry]` through `tes-engineering-discipline`. The new skill exists to make explicit invocation deterministic, not to replace the existing auto-activation path.

### Cursor

Cursor does not need a separate skill folder unless the adapter gains native skills. Cursor rules should only mention `/flash-fry` as compatible TES intent if command-trigger routing is updated.

## Skill Shape

Each new Tilly skill must follow the skill construction contract:

```text
SKILL.md routes and operates.
docs/CONTRACT-HISTORY.md preserves why.
agents/openai.yaml exposes the trigger.
```

Required source paths:

```text
src/adapters/codex/skills/flash-fry/SKILL.md
src/adapters/codex/skills/flash-fry/docs/CONTRACT-HISTORY.md
src/adapters/codex/skills/flash-fry/agents/openai.yaml
src/adapters/claude/skills/flash-fry/SKILL.md
src/adapters/claude/skills/flash-fry/docs/CONTRACT-HISTORY.md
src/adapters/claude/skills/flash-fry/agents/openai.yaml
```

The `SKILL.md` files should be lean. They should point to Mantra Gate helpers instead of restating the full Mantra Gate contract.

## Required Installer Behavior

After implementation and installation, a target project should contain:

```text
.claude/skills/flash-fry/SKILL.md
.agents/skills/flash-fry/SKILL.md
```

The installer must not write global skill locations to satisfy this feature. The installer must not require plugin marketplace installation.

If a project already has a user-owned `/flash-fry` command, TES must not overwrite it blindly. It should report a conflict and route through the project-local skill when possible.

If TES intentionally supports `/tes-flash-fry`, the installer should implement it as an alias surface, not by making `tes-flash-fry` the canonical skill. On Claude Code, the simplest alias is a project-local command file that forwards intent to the `flash-fry` skill, while leaving `.claude/skills/flash-fry/**` as the source of behavior.

## Oracle Plan

The smallest useful oracle sequence is:

1. Skill source validation:

   ```bash
   python3 $CODEX_HOME/skills/.system/skill-creator/scripts/quick_validate.py \
     src/adapters/codex/skills/flash-fry
   python3 $CODEX_HOME/skills/.system/skill-creator/scripts/quick_validate.py \
     src/adapters/claude/skills/flash-fry
   ```

2. Trigger and materialization checks:

   ```bash
   python3 scripts/command_trigger_oracle.py --self-test
   python3 scripts/materialize_adapter.py all --check
   python3 scripts/validate_reference_package.py
   ```

3. Mantra Gate helper checks:

   ```bash
   python3 scripts/mantra_gate.py --self-test
   python3 scripts/mantra_gate.py emit-marker
   python3 scripts/mantra_gate_adoption_oracle.py --self-test
   ```

4. Full package closure before sync:

   ```bash
   npm run commit:check
   ```

5. If released, installer channel certification:

   ```bash
   npm run release:check
   ```

## Acceptance Criteria

The implementation is complete only when all criteria are true:

- `flash-fry` exists in Codex and Claude source adapters.
- Materialized adapter output contains both project-local installed skills.
- `/flash-fry` is documented as the native project-local skill invocation.
- `/tes-flash-fry` is documented only as an optional alias if materialized.
- `/tes:flash-fry` is normalized as operator intent, not promised as a native skill namespace.
- No source claims that a global plugin is required for Flash-Fry.
- The new skill delegates to Mantra Gate helpers.
- Command trigger oracle covers `/flash-fry` and any compatibility aliases that are intentionally materialized.
- Reference package validation requires the new files.
- Full `commit:check` passes.
- If release identity changes, `tes-bump` requires and validates the bump.

## Failure Modes To Prevent

| Failure | Prevention |
|---------|------------|
| Runtime tries `tilly:flash-fry` and fails | Classify as plugin-namespace routing mismatch and route project-local `/flash-fry`. |
| Flash-Fry becomes a second gate | Skill delegates to Mantra Gate helpers only. |
| Claude depends on global plugin install | Spec requires `.claude/skills/**` project-local source. |
| Codex auto-marker hides missing explicit command | Explicit skill and trigger oracle cover invocation. |
| Target project defines behavior | TES package source remains the source of truth. |
| Installer overwrites user command | Conflict is reported instead of blind overwrite. |

## Release Identity

This is delivered behavior. Implementing this spec requires a patch version bump and correlated release surfaces if the package is synced.

The release cannot be called installer-certified until:

1. the versioned public bundle is regenerated;
2. the fixed Git ref exists remotely;
3. `npm run release:check` passes against the new ref.

## Open Questions

1. Should `tilly:flash-fry` be documented as an invalid plugin namespace for project-local installs or as an optional host-specific alias when a future host supports namespaces?
2. Should `/flash-fry audit` run `--audit-history` by default, or should audit require an explicit phrase such as `/flash-fry audit-history`?
3. Should the skill emit only the marker for routine invocations, or always include a one-line status?

Until these are answered, the conservative default is:

- project-local `flash-fry`;
- `/flash-fry` as the native trigger;
- `tilly:flash-fry` treated as a routing mismatch;
- audit-history only on explicit request.
