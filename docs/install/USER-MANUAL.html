··— Tilly Engineer Skills

A field manual for governed coding agents.

A practical guide for installing, updating and auditing Tilly Engineer Skills in real projects — without blind file copying.

00

Overview

Tilly installs a governance mesh for coding agents. Codex, Claude Code and Cursor route to the same durable project source: docs/agents/**. Real governance lives in versioned Markdown.

How you use it

The normal user interface is an intent typed in the agent window — not a shell command. Type /tes-init to install or /tes-update to update; the active agent selects and runs the smallest safe oracle.

What stays yours

Existing AGENTS.md, CLAUDE.md, CURSOR.md and Cursor rules are first backed up in .tes/bk/<timestamp>/. Tilly then installs a clean runtime and recovers useful local semantics into docs/agents/**.

Operator contract

This manual is for users installing, auditing and rolling back Tilly. The agent-side contract — triggers, oracles, gates, return states, schemas — lives in docs/install/AGENT-MANUAL.md.

01

Quick install

  1. Open the target project in Codex, Claude Code or Cursor.
  2. Type /tes-init in the agent window for first setup, or /tes-update if Tilly is already installed. If your runtime does not interpret the trigger, paste the installer prompt from the README or docs/install/MINI-PROMPT.md.
  3. Let Step Zero finish before approving any edit. Step Zero records a baseline so rollback stays clean.
  4. Choose a route when prompted: current, codex, claude, cursor, all, mcp or audit.
  5. Read the final report before asking for commit or push.

05

Cortex

Cortex is the project's compiled memory, written in plain Markdown under docs/agents/cortex/. Every claim is backed by an evidence link to a source file. SQLite caches accelerate recall but never replace Markdown as the source of truth.

You can read Cortex like any other repository document, review it in pull requests, and roll it back with Git. The agent never writes to memory without explicit authorization and an evidence link.

Backend selection, cell schema, and write rules are part of the agent contract — see AGENT-MANUAL.md §8.

06

What the agent runs

Inside the agent window you do not type shell commands. You type intent — /tes-init, /tes-update, /tes-align, /tes-open-obsidian — and the agent translates that into a sequence of small, scoped oracles.

You only need to know two things:

  1. Step Zero runs first. Before any write, the agent records a baseline and verifies its own toolchain. If a required capability is missing, the run halts and reports it instead of guessing.
  2. The final report is the contract. Read it before approving commit or push. If status is anything other than success, ask the agent to explain or roll back.

The full oracle inventory — Python scripts, npm run wrappers, validation gates, probe contracts — lives in AGENT-MANUAL.md §9. Reach for it when reviewing a run, debugging a blocked oracle, or auditing what the agent invoked.

On /tes-update, the agent first repairs the bundled helpers — a helper-only Layer Zero pass — before touching adapter or MCP configuration. The update probe then reconciles installed and cloud versions and checks helper contract parity (the user manual notes the same step in Portuguese as versão instalada e versão na nuvem com paridade de contrato dos helpers) before any GO decision.

The downloaded installer bundle is staging cache. TES keeps it under .tes/setup/<version>/, ensures that path is ignored through the target repository's local Git exclude, and does not treat extracted ZIP payloads as project knowledge to commit.

08

Field Reports

Field Reports lets Tilly learn from real installations. It is active by default and only captures sanitized operational facts. It never sends your code, diffs, prompts, file contents, secrets, personal data, absolute paths, raw stack traces, branch names, or remote URLs.

Reports drain only during git push, only when the GitHub CLI, authentication and network are all available. If any link is missing, nothing leaves your machine and the push proceeds normally.

Two privacy gates apply: local sanitization on capture, and GitHub-side schema checks plus a quarantine workflow on receipt.

To opt out, type this in the agent window:

/tes:field-reports:disable

To re-enable later:

/tes:field-reports:enable

The full transport pipeline — pre-push hook, outbox state, receiver schema — is documented in AGENT-MANUAL.md §11.

09

Obsidian

Obsidian is an optional visual workbench for browsing the agent mesh and Cortex memory. It is not required, and Tilly never creates or edits .obsidian/**. Type /tes-open-obsidian in the agent window to open docs/agents/ as a vault.

  1. Open docs/agents/ as the vault to see agent docs, evidence, and Cortex together.
  2. Open docs/agents/cortex/ alone when you want a focused memory workspace.
  3. Use Graph View to spot isolated cells and weak hubs.
Obsidian settingRecommended use

A pretty graph is not proof. Trust Git diffs, the agent's audit run, and explicit evidence — not visual density.

10

Final report

Every run ends with a short Convergence Report in chat. Read it before approving commit or push. It tells you what changed, which .tes/bk/<timestamp> backup was created, what was recovered, whether all gates passed, and which rollback is available.

The chat report covers status, source snapshot freshness, changed surfaces, root context gate, installed helper set, helper contract parity, Field Reports state, gates, limits, and a rollback summary. Backup files are rollback artifacts, not new TES surfaces.

A few signals worth recognizing:

  • RECOVERED — previous AGENTS.md, CLAUDE.md or Cursor rules were backed up and useful semantics were recovered into docs/agents/**.
  • STALE_SOURCE — the package commit is behind or unrelated to the latest Tilly release; you were certified against the recorded snapshot. Current public bundles whose source commit is an ancestor of the distribution commit report PASS, not stale.
  • NEEDS_REVIEW — something blocked but the run preserved evidence; review before continuing.

The full report-field contract is in AGENT-MANUAL.md §14.

Rollback

Use the baseline recorded during Step Zero.

git reset --hard <baseline-head>
git revert <install-commit>

Informe final

Cada ejecución termina con un Convergence Report corto en el chat. Léelo antes de aprobar commit o push. Te dice qué cambió, qué backup .tes/bk/<timestamp> fue creado, qué se recuperó, si todos los gates pasaron y qué rollback está disponible.

Algunas señales que conviene reconocer:

  • RECOVERED — tus AGENTS.md, CLAUDE.md o reglas de Cursor anteriores fueron respaldados y la semántica útil fue recuperada en docs/agents/**.
  • STALE_SOURCE — el commit del paquete está atrás o no está relacionado con el último release de Tilly; los bundles públicos actuales cuyo commit fuente es ancestro del commit de distribución reportan PASS, no stale.
  • NEEDS_REVIEW — algo se bloqueó pero el run preservó evidencia; revisa antes de continuar.

09

Obsidian

O Obsidian é uma bancada visual opcional para navegar a malha do agente e a memória Cortex. Ele não é obrigatório, e a Tilly nunca cria ou edita .obsidian/**. Digite /tes-open-obsidian na janela do agente para abrir docs/agents/ como vault.

  1. Abra docs/agents/ como vault para ver docs de agente, evidências e Cortex juntos.
  2. Abra docs/agents/cortex/ apenas quando quiser um espaço focado de memória.
  3. Use o Graph View para identificar cells isoladas e hubs fracos.

10

Relatório final

Toda execução termina com um Convergence Report curto no chat. Leia antes de aprovar commit ou push. Ele diz o que mudou, qual backup .tes/bk/<timestamp> foi criado, o que foi recuperado, se todos os gates passaram e qual rollback está disponível.

O relatório no chat cobre status, frescor do snapshot fonte, superfícies alteradas, gate de contexto raiz, helper set instalado, paridade de contrato dos helpers, estado do Field Reports, gates, limites e resumo de rollback. Backups são artefatos de rollback, não novas superfícies TES.

Alguns sinais que vale reconhecer:

  • RECOVERED — seus AGENTS.md, CLAUDE.md ou regras do Cursor anteriores foram salvos e a semântica útil foi recuperada em docs/agents/**.
  • STALE_SOURCE — o commit do pacote está atrás ou não se relaciona ao release mais recente da Tilly; você foi certificado contra o snapshot registrado. Bundles públicos atuais cujo commit fonte é ancestral do commit de distribuição reportam PASS, não stale.
  • NEEDS_REVIEW — algo bloqueou mas o run preservou evidência; revise antes de continuar.

O contrato completo de campos do relatório está em AGENT-MANUAL.md §14.
