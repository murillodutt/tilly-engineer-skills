Guia operacional

Manual de uso do Tilly Engineer Skills

Um guia prático para instalar, atualizar, auditar e operar o Tilly como context mesh assistido em projetos reais, sem cópia cega de arquivos e sem transformar documentação em enciclopédia.

Visão geral

Tilly Engineer Skills instala um mesh de governança para agentes de código. O objetivo é fazer Codex, Claude Code e Cursor seguirem o mesmo contrato local do projeto: pensar antes de editar, manter o escopo simples, tocar só no necessário e verificar antes de fechar.

A regra central é simples: docs/agents/** é a fonte durável. Arquivos como AGENTS.md, CLAUDE.md e regras do Cursor são bootloaders finos que apontam para essa fonte.

filesystem-first auditável Codex Claude Code Cursor Cortex

Quando usar

Situação Use Tilly para
Projeto novo Criar um mesh mínimo com bootloader do runtime ativo.
Projeto com instruções espalhadas Retrofit: preservar regras locais e movê-las para docs/agents/**.
Projeto já com Tilly Atualizar por convergência sem reinstalar nem sobrescrever governança local.
Auditoria Inspecionar estado, conflitos e rotas sem escrever arquivos.

Instalação rápida

  1. Abra o projeto alvo no Codex, Claude Code ou Cursor.
  2. Garanta que o projeto esteja em Git.
  3. Cole o prompt abaixo na janela do agente.
  4. Escolha a rota quando o instalador pedir.
  5. Leia o relatório final antes de commitar ou publicar.
Install Tilly Engineer Skills as an assisted context mesh, not as blind file
copying.

Read and follow this raw installer spec:

https://raw.githubusercontent.com/murillodutt/tilly-engineer-skills/main/docs/install/ASSISTED-CONTEXT-INSTALLER.prompt.md

Run in quiet installer mode: show compact progress, blockers and the final
certification report only. When navigation is required, load the runtime
navigation library from the spec, use native structured cards only when the
current runtime safely supports them, otherwise render command navigation.
Ask for a route command such as current, codex, claude, cursor, all, or audit.
Do not display internal reasoning, scratch YAML or long inventories.

Start by detecting the current IDE/runtime and classifying this project as new,
existing, or meshed. If Tilly is already meshed, treat this as an assisted
update/convergence run, not a reinstall. Use the detected IDE as the default
adapter. Ask me for a route command only where the spec requires one. Preserve
local project governance, move durable agent context into docs/agents/** when
needed, create or update the compiled docs/agents/cortex/** Cortex layer, keep
AGENTS.md, CLAUDE.md and Cursor rules as thin runtime bootloaders, and finish
with the certification report required by the spec.

Before installation edits, run Step Zero from the spec: inspect Git status and
offer a local baseline commit if the working tree is dirty. At the end, tell me
how to undo the installation with Git. Do not push, amend, tag, publish, install
dependencies, overwrite files, or change remotes unless I explicitly ask after
reviewing the certification report.

O instalador não deve fazer push, tag, publish, alteração de remoto ou instalação de dependências sem autorização explícita depois do relatório de certificação.

Rotas

Quando o instalador pedir uma rota, responda com um comando curto. A rota decide quais superfícies serão criadas, atualizadas ou auditadas.

Comando Uso recomendado
current Instalar ou atualizar apenas o runtime atual. É a escolha padrão.
codex Criar ou atualizar AGENTS.md e skill Codex.
claude Criar ou atualizar CLAUDE.md e assets Claude.
cursor Criar ou atualizar regras em .cursor/rules/**.
all Preparar Codex, Claude e Cursor no mesmo projeto.
audit Inspecionar sem modificar arquivos.

Estados do projeto

new

Projeto sem orientação de agente relevante. O instalador cria o mesh mínimo.

existing

Projeto já tem regras, docs ou scripts próprios. O instalador faz retrofit, não overwrite.

meshed

Tilly já existe. A execução vira update/convergence, com mudanças cirúrgicas.

uncertain

O agente não conseguiu classificar com confiança. Ele deve reportar o bloqueio.

Cortex

Cortex é a memória compilada do projeto. Ele não é banco de dados, não é RAG escondido e não é saída solta de LLM. A memória vive em arquivos versionados. 

docs/agents/cortex/
  CONTRACT.md
  MAP.md
  TRAIL.md
  LINKS.md
  sources/
  cells/
.tilly/cortex/
  recall.sqlite
Camada Regra
sources/** Fontes brutas imutáveis depois do import.
cells/** Conhecimento compilado com ## Claim e ## Evidence.
MAP.md Mapa de navegação que agentes leem primeiro.
TRAIL.md Linha do tempo append-only da evolução do Cortex.
LINKS.md Relações principais entre células, fontes e decisões.
recall.sqlite Índice derivado, apagável e reconstruível. Nunca é memória.

Obsidian

O Cortex é visível no Obsidian porque é Markdown comum. Abra a raiz do projeto ou docs/agents/cortex/ como vault para navegar por links e grafo.

O instalador não cria nem edita .obsidian/**. Obsidian é superfície visual, não fonte de verdade e não dependência de certificação.

Atualização

Use o mesmo prompt da instalação. Se o projeto já estiver meshed, o agente deve atualizar por convergência.

O que uma atualização saudável faz

  • detecta versão atual e superfícies presentes;
  • preserva regras locais;
  • não reinstala nem sobrescreve bootloaders sem necessidade;
  • atualiza contratos e Cortex quando houver delta real;
  • roda os oracles seguros antes do relatório final.

Certificação

Todo fluxo termina com um Tilly Context Mesh Convergence Report. O relatório precisa dizer o que foi feito, o que foi preservado, quais comandos rodaram e qual é a claim final.

O relatório final sempre deve expor este manual por link web ou path local. Abrir o navegador automaticamente é conveniência opcional, não requisito de certificação.

Claim Significado
GO meshed Mesh criado, retrofitted ou atualizado; oracles locais passaram.
GO committed Usuário aprovou commit depois do relatório.
GO published Usuário aprovou push ou publicação depois do commit.
NEEDS_REVIEW Há decisão humana pendente, conflito ou oracle não executado.
NO-GO O instalador encontrou falha que impede certificação.

Rollback

Antes de editar, o instalador roda Step Zero. Se a árvore estiver suja, ele oferece um commit local de baseline. Isso torna o rollback claro. 

# Descartar mudanças após o baseline registrado
git reset --hard <baseline-head>

# Desfazer uma instalação commitada separadamente
git revert <install-commit>

Não rode rollback automaticamente. O agente deve explicar o comando e esperar a decisão do usuário.

Problemas comuns

Sintoma Resposta prática
O agente quer sobrescrever instruções existentes. Pare. O fluxo correto é preservar e fazer retrofit.
O projeto já tem Tilly, mas o agente quer reinstalar. Reforce o estado meshed e peça update/convergence.
Cortex virou resumo solto. Rode audit. Cells precisam de Claim e Evidence.
SQLite sumiu ou quebrou. Rode rebuild. O banco é cache derivado.
Obsidian mostra links estranhos. Verifique LINKS.md, wikilinks e paths explícitos.

Manutenção

Para operadores do pacote, o ciclo saudável é pequeno: mudar um corte, rodar gate, commitar semanticamente e só publicar com autorização. 

npm run commit:check
python3 scripts/cortex.py --self-test
python3 scripts/materialize_adapter.py all --check
python3 scripts/adapter_parity_readiness.py

Git é a história real. TVer versiona contratos vivos. Evidência versionada sustenta certificações. O manual aponta o caminho; os gates provam o estado.
