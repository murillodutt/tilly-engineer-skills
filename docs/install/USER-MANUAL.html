··— Tilly Engineer Skills

A field manual for governed coding agents.

A practical guide for installing, updating and auditing Tilly Engineer Skills in real projects — without blind file copying.

00

Overview

Tilly installs a governance mesh for coding agents. Codex, Claude Code and Cursor route to the same durable project source: docs/agents/**. Bootloaders stay thin. Real governance lives in versioned Markdown.

Note

Files such as AGENTS.md, CLAUDE.md and Cursor rules should stay thin bootloaders. Real governance lives in docs/agents/**.

Execution model

The executor is the active coding agent inside the current IDE or runtime window. Skills, rules and bootloaders route behavior; Python scripts and npm run ... entries are deterministic oracles the agent invokes when local tool access exists; hooks are Git-event gates; MCP is a read-only Cortex access surface. If a runtime cannot execute an oracle, the agent must report it as blocked or skipped instead of claiming certification.

01

Quick install

  1. Open the target project in Codex, Claude Code or Cursor.
  2. Type /tilly:init when the Tilly init skill/rule is available. Otherwise paste the installer prompt from the README or docs/install/MINI-PROMPT.md.
  3. Let the installer run Step Zero before any edit.
  4. Choose a route: current, codex, claude, cursor, all, mcp or audit.
  5. Read the final report before asking for commit or push.

04

Cortex

Cortex is the project's compiled memory. Memory lives in versioned Markdown; SQLite is only a derived index.

docs/agents/cortex/
  CONTRACT.md
  MAP.md
  TRAIL.md
  LINKS.md
  sources/
  cells/
.tilly/cortex/recall.sqlite

Cells must include ## Claim and ## Evidence. sources/** is immutable after import. learn proposes without writing; apply --yes writes only authorized, evidenced cells and refreshes recall after audit.

CLI

These commands are agent-callable oracles, not the normal user interface. In normal use, the active IDE agent invokes them from the current context window; humans run them directly only for package maintenance, debugging or certification review. Use the Tilly package root for package checks, or pass --target when operating on another project or vault. tilly_init.py recertifies package health, scans the target project, writes docs/agents/PROJECT-REGISTER.md and stores a full manifest under docs/agents/evidence/**.

Prefer intent shortcuts: /tilly:init, /tilly:cortex, /tilly:mcp, /tilly:doctor, /tilly:adapter and /tilly:bench. The agent translates them into the smallest safe oracle. MCP

The MCP server is read-only. The installer activates it through project-scoped config for the selected runtime route, without turning MCP, SQLite or the LLM into memory.

python3 scripts/install_mcp.py --target /path/to/project --adapter codex --yes
python3 scripts/install_mcp.py --target /path/to/project --adapter all --yes
python3 scripts/install_mcp.py --self-test
python3 scripts/install_smoke.py --route mcp
python3 scripts/cortex_mcp.py --target /path/to/project-or-vault
python3 scripts/cortex_mcp.py --self-test
npm run mcp:self-test
npm run cortex:mcp:self-test
RuntimeProject-scoped config
Codex.codex/config.toml
Claude Code.mcp.json
Cursor.cursor/mcp.json
ToolBehavior
cortex_verifyValidates the Cortex structure and contract.
cortex_auditFinds broken links, missing evidence, or loose cells.
cortex_recallSearches through SQLite FTS5 and falls back to rg.
cortex_read_cellReads one cell directly from cells/**.
cortex_absorb_planProduces a no-write absorb plan.
cortex_reflectGenerates a no-write memory and curation proposal.

Write-capable MCP tools are intentionally outside this version. Activation writes only .tilly/bin/** and project-scoped runtime config; it must not touch global MCP config, secrets, hooks or sources/**.

07

Obsidian

Use Obsidian as the visual workbench for Cortex, not as the authority layer. The installer never creates or edits .obsidian/**; the user configures Obsidian locally and Cortex remains plain Markdown.

  1. Open the project root as the vault when you want to see agent docs, evidence and Cortex together.
  2. Open docs/agents/cortex/ as the vault when you want a focused memory workspace.
  3. Set the attachment folder to sources/assets/ when saving local images or clipped article assets.
  4. Keep source citations as repository paths such as sources/article.md; use wikilinks mainly between cells.
  5. Use Graph View to inspect hubs, isolated cells and relationship density after each meaningful absorb/apply cycle.
  6. Run python3 scripts/cortex.py audit --target /path/to/project-or-vault before trusting the graph.
Obsidian settingRecommended use
Vault rootProject root for full context, or docs/agents/cortex/ for focused memory.
Files and linksKeep Markdown links readable outside Obsidian; avoid Obsidian-only proof.
AttachmentsUse sources/assets/ for local images and clipped media.
Graph filtersFocus on cells/, MAP.md and LINKS.md; hide transient or generated files.
Community pluginsOptional only. Dataview, Canvas and Marp must not be required for certification.

A healthy Cortex graph should show cells connected by claims and evidence. A pretty graph is not proof: audit, Git diff and explicit evidence are the certification layer.

Vista general

Tilly instala un mesh de gobernanza para agentes de código. Codex, Claude Code y Cursor apuntan a la misma fuente durable del proyecto: docs/agents/**. Los bootloaders se mantienen finos. La gobernanza real vive en Markdown versionado.

Nota

Archivos como AGENTS.md, CLAUDE.md y las reglas de Cursor deben ser bootloaders finos. La gobernanza real vive en docs/agents/**.

Modelo de ejecución

El ejecutor es el agente de código activo dentro de la ventana actual del IDE o runtime. Skills, reglas y bootloaders enrutan el comportamiento; los scripts Python y las entradas npm run ... son oráculos deterministas que el agente invoca cuando existe acceso local a herramientas; los hooks son gates de eventos Git; MCP es una superficie Cortex de solo lectura. Si un runtime no puede ejecutar un oráculo, el agente debe reportarlo como bloqueado o omitido en vez de certificarlo.

01

Instalación rápida

  1. Abre el proyecto objetivo en Codex, Claude Code o Cursor.

  3. Deja que el instalador ejecute Step Zero antes de editar.
  4. Elige una ruta: current, codex, claude, cursor, all, mcp o audit.
  5. Lee el informe final antes de pedir commit o push.

Use a raiz do pacote Tilly para checks do pacote, ou --target para operar em outro projeto ou vault. tilly_init.py recertifica a saúde do pacote, varre o projeto alvo, escreve docs/agents/PROJECT-REGISTER.md e guarda manifestos com timestamp em docs/agents/evidence/**.

Prefira atalhos de intenção: /tilly:init, /tilly:cortex, /tilly:mcp, /tilly:doctor, /tilly:adapter e /tilly:bench. O agente traduz isso para o menor oráculo seguro. Obsidian

Use o Obsidian como bancada visual do Cortex, não como camada de autoridade. O instalador nunca cria nem edita .obsidian/**; o usuário configura o Obsidian localmente e o Cortex continua sendo Markdown comum.

  1. Abra a raiz do projeto como vault quando quiser ver docs de agente, evidências e Cortex juntos.
  2. Abra docs/agents/cortex/ como vault quando quiser um espaço focado de memória.
  3. Configure a pasta de anexos como sources/assets/ para imagens locais ou assets de artigos recortados.
  4. Mantenha citações de fontes como paths do repositório, por exemplo sources/article.md; use wikilinks principalmente entre cells.
  5. Use o Graph View para revisar hubs, cells isoladas e densidade de relações depois de cada ciclo relevante de absorb/apply.
  6. Rode python3 scripts/cortex.py audit --target /path/to/project-or-vault antes de confiar no grafo.
Ajuste do ObsidianUso recomendado
Vault rootRaiz do projeto para contexto completo, ou docs/agents/cortex/ para memória focada.
Files and linksMantenha links Markdown legíveis fora do Obsidian; evite prova que só existe no Obsidian.
AttachmentsUse sources/assets/ para imagens locais e mídia recortada.
Graph filtersFoque em cells/, MAP.md e LINKS.md; esconda arquivos transitórios ou gerados.
Community pluginsApenas opcionais. Dataview, Canvas e Marp não devem ser exigidos para certificação.

Um grafo Cortex saudável mostra cells conectadas por claims e evidências. Um grafo bonito não é prova: audit, Git diff e evidência explícita são a camada de certificação.
