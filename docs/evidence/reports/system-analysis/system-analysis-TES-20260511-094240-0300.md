# System Analysis TES - 20260511-094240-0300

## Resumo Executivo

Este raio X analisou o pacote `tilly-engineer-skills` como sistema de
engenharia, pacote de adapters, camada documental governada e artefato Git.

Resultado principal: o projeto está estruturalmente saudável, com Git limpo,
`main` alinhada a `origin/main`, TDS validado, materialização passando,
plataforma certificada localmente e `npm run commit:check` verde na varredura
do subagente de Git/oráculos.

O estado não é "sem risco". Há achados que merecem fila de produto:

- Um helper instalado, `.tes/bin/materialize_adapter.py`, é entregue ao target,
  mas falha fora do pacote fonte porque espera `src/adapters/**`.
- A paridade de comandos é certificada em parte por grupos agregados de arquivos;
  isso pode esconder drift em uma superfície individual.
- O manual expõe aliases de Field Reports `:disable` e `:enable` que não aparecem
  no conjunto certificado de aliases dos adapters.
- O oracle de plataforma rotula a superfície Cursor como `skill`, embora o
  contrato diga que Cursor v1 usa regras e tem certificação comportamental
  apenas estrutural.
- Field Reports é comportamento entregue com hook local e transporte GitHub
  condicionado; deve ser tratado como superfície operacional, não como simples
  validador interno.

Decisão desta análise: `GO` para integridade estrutural e governança atual do
pacote; `NEEDS_REVIEW` para os riscos de helper instalado, cobertura de aliases
e rotulagem Cursor antes de qualquer claim comercial mais forte.

## Achados De Risco

### R1 - Helper Instalado Com Falha Fora Do Pacote Fonte

Status: `NEEDS_REVIEW`.

`scripts/install_mcp.py:25-40` instala `materialize_adapter.py` em `.tes/bin/**`.
`docs/install/AGENT-MANUAL.md:382-388` lista esse helper como instalado.
Entretanto, `scripts/install_mcp.py:381-405` valida existência e
`cortex_mcp.py --self-test`, enquanto `scripts/install_mcp.py:444-452` nao inclui
`materialize_adapter.py` nos self-tests instalados.

Teste real em target temporário:

```text
python3 .tes/bin/materialize_adapter.py all --check
FileNotFoundError: missing source: src/adapters/claude/CLAUDE.md
```

Impacto provável: `/tes-adapter` ou agentes em target instalado podem tratar o
helper como funcional, mas ele depende da árvore fonte do pacote.

Recomendação: remover esse helper do conjunto instalado ou torná-lo
installed-aware; em ambos os casos, adicionar um self-test instalado que prove o
contrato real.

### R2 - Cobertura Agregada Em Command Trigger Oracle

Status: `NEEDS_REVIEW`.

`scripts/command_trigger_oracle.py:99-131` valida grupos de fontes, e
`scripts/command_trigger_oracle.py:177-186` concatena arquivos do grupo antes de
checar termos. Isso é útil para cobertura geral, mas pode permitir que um arquivo
individual derive enquanto outro arquivo do grupo mantém o termo.

Exemplo: `docs/adapters/CODEX.md:66-83` lista vários shortcuts, mas omite
`/tes-curate` e `/tes-field-reports` nessa tabela, enquanto
`docs/install/COMMAND-TRIGGERS.md:17-20` declara ambos como triggers
compartilhados.

Recomendação: validar arquivos de docs/adapters individualmente para os
shortcuts que eles declaram cobrir, ou tornar explícito que cada adapter guide é
parcial e aponta para `COMMAND-TRIGGERS.md` como matriz completa.

### R3 - Alias Field Reports No Manual Fora Do Alias Certificado

Status: `NEEDS_REVIEW`.

`docs/install/AGENT-MANUAL.md:51` documenta
`/tes:field-reports:disable` e `/tes:field-reports:enable`.
`docs/install/USER-MANUAL.html:1595-1599` expõe os mesmos prompts ao usuário.
Mas `scripts/command_trigger_oracle.py:31-48` certifica apenas
`/tes:field-reports`, sem os sufixos `:disable` e `:enable`.

Recomendação: adicionar os aliases ao oracle e aos adapters, ou reduzir o manual
para um único trigger certificado com intenção textual de disable/enable.

### R4 - Cursor Rotulado Como Skill Pelo Oracle De Plataforma

Status: `NEEDS_REVIEW`.

`docs/adapters/ADAPTER-CAPABILITY-MATRIX.md:69-70` diz que TES v1 usa regras para
Cursor e não claim de plugin skills. Ainda assim,
`scripts/platform_surface_oracle.py:238-240` registra uma superfície Cursor
`skill` como `certified`, usando `tes-runtime-capabilities.mdc` como evidência.

Impacto provável: relatórios downstream podem transformar "rule-based command
routing" em "Cursor skill certificado", contrariando a própria matriz.

Recomendação: renomear essa superfície para `rule-router`, `command-router` ou
similar, mantendo Cursor como estrutural.

### R5 - Field Reports Como Superficie Entregue Com Fronteira Externa

Status: `WATCH`.

`docs/install/INSTALL.md:153-162` declara Field Reports ativo por padrão, com
`pre-push`, `.tes/field-reports/**`, transporte fake certificado e publicação
real dependente de `gh`, autenticação e rede. `scripts/field_reports.py:23`
define destino `murillodutt/tilly-engineer-skills`, e
`scripts/field_reports.py:689-699` chama `gh issue create`.

Recomendação: manter Field Reports classificado como comportamento
entregue/adotante com efeito externo condicionado. Ele não deve ser descrito
como mero gate maintainer.

### R6 - Pressao De Tamanho Documental

Status: `WATCH`.

`npm run commit:check` passou, mas o subagente de Git/oráculos observou avisos de
documentos perto do limite:

- `docs/index.html`

- `docs/install/AGENT-MANUAL.md`

Recomendação: antes de novas expansões, modularizar ou reduzir essas superfícies
em vez de aumentar limites.
